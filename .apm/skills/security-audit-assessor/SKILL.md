# Security Audit Assessor

このスキルは、リポジトリや付随文書を対象に、`IPA + NIST` のハイブリッド観点で静的セキュリティ監査を行う。
通常はローカルの `JSON index + mode別 mapping` だけを読み、根拠補強が必要なときだけ公式 URL に降りる。NIST や IPA の原文を毎回読む前提にはしない。

## 監査モード

最初に次のいずれかを選ぶ。

- `system`: Webアプリ、API、DB、設定、IaC、CI/CD、秘密情報、認証認可、ログ、バックアップなど技術実装を監査する。
- `governance`: 規程、運用手順、権限棚卸、委託先管理、インシデント対応、教育、監査証跡など管理統制を監査する。
- `combined`: `system` と `governance` を順に実施し、`overlap_group` が同じ指摘を統合した監査報告書を出す。

ユーザー指定がなければ、対象がコード中心なら `system`、文書中心なら `governance`、両方見たい意図があるなら `combined` を選ぶ。

## 最初に読むファイル

1. `references/index/documents.json`
2. `references/index/README.md`
3. 監査モードに応じて次のどちらか、または両方
   - `references/mappings/system-controls.json`
   - `references/mappings/governance-controls.json`
4. 必要に応じて次を追加
   - `references/index/topics.json`
   - `references/index/controls.json`
   - `references/index/aliases.json`
5. `references/question-triage.md`
6. 出力直前に `references/report-template.md`

通常は `JSON index` と `mapping` で対象 section を引き、必要な section だけ `references/corpus/*.toon` から読む。`Critical` または `High` の指摘で根拠補強が必要な場合だけ、`security-standards.json` の `canonical_url` を使って公式原文を見に行く。

## 実行手順

1. 監査対象を確認する。
   - システム名
   - 対象環境
   - 扱うデータ種別
   - 外部公開有無
   - クラウド/オンプレ構成
2. 判明している情報だけで仮の前提を置く。
3. 監査モードを決める。
4. 対象に応じてコード、設定、IaC、CI/CD、README、運用手順、ポリシー文書を収集する。
5. 選択したモードの `mapping JSON` から監査項目を読み、`related_sections` を起点に証跡を探す。
6. 監査項目ごとに `適合` / `要改善` / `不適合` / `要確認` で判定する。
7. リポジトリや付随文書にない事実で判定に必要なものがあれば、不足情報質問を段階的に出す。
8. 回答があれば判定を更新し、回答がなくても監査を完了する。
9. `combined` の場合は `overlap_group` が同じ指摘を統合し、system/governance の両面から根拠をまとめる。
10. 最後に監査報告書形式で結果を出力する。

## 質問ルール

質問は `references/question-triage.md` の基準で行う。

- 原則として自由入力ではなく、`2-4個の選択肢 + 不明/未確認` の選択式で聞く。
- まずリポジトリ、設定、IaC、README、運用文書と `references/index/*.json` を読み、そこに書かれていない事実だけを聞く。
- 質問は段階式で出し、まず「監査上もっとも影響範囲が広い前提」から確定する。
- 選択肢は、既存資料から見て尤度が高い順、または監査影響が大きい順に並べる。
- 各質問には、選ぶべき単位が分かる短い選択肢ラベルを付ける。
- 自由入力の質問は、選択肢では監査不能な場合に限る。
- 各質問には「どの監査項目に影響するか」を書く。
- 既存資料から推測可能なことは質問しない。
- 回答がなくても停止しない。
- 未確認のまま残る項目は `要確認` にする。

質問例:

- 質問: `本番環境の公開範囲はどれですか`
  選択肢: `1. インターネット公開 2. 社内/VPN限定 3. 開発環境のみ 4. 不明`
  影響項目: `ネットワーク境界, WAF, 公開面の防御`
- 質問: `扱うデータ種別はどれに近いですか`
  選択肢: `1. 個人情報あり 2. 機密業務データ中心 3. 機微情報なし 4. 不明`
  影響項目: `データ保護, 暗号化, アクセス制御, ログ方針`

## 選択式質問の作り方

質問を出すときは、次の順序で組み立てる。

1. 先に暫定前提を1行で置く。
2. その前提を確定するための質問を1つ出す。
3. 回答候補を `2-5個 + 不明/未確認` に絞る。
4. 各選択肢がどの監査項目に効くかを簡潔に示す。

質問フォーマット:

- 暫定前提: `...`
- 質問: `...`
- 選択肢:
  - `1. ...`
  - `2. ...`
  - `3. ...`
  - `4. ...`
  - `5. 不明/未確認`
- 影響項目: `監査項目A, 監査項目B`
- 未回答時の扱い: `要確認のまま報告する`

複数質問が必要でも、毎回「いまの判定に必要な最小限」だけを出し、ユーザーが「番号で選ぶだけ」で返せる形に保つ。回答次第で不要になった後続質問は出さない。

## 判定基準

- `適合`: 証跡があり、要求を満たしていることを確認できる。
- `要改善`: 実装や運用はあるが、網羅性、統制、継続性の不足がある。
- `不適合`: 明確な欠落や危険な実装が確認できる。
- `要確認`: 証跡不足または資料不足で断定できない。

推測で `適合` にしてはいけない。証跡が弱い場合は `要確認` に落とす。

## 根拠の使い方

- 既定では逐語引用しない。
- 監査結果には、文書名、版、framework別カテゴリまたはコントロールファミリ、公式 URL を残す。
- `reference_map` は `ipa` / `nist_csf` / `nist_sp_800_53` を分けて扱い、混在させない。
- 根拠補強で公式原文を見た場合だけ、必要最小限の短い抜粋を付けてよい。
- URL を確認できない場合は、ローカル index 情報に基づいて報告しつつ `原典未確認` と明記する。

## 優先順位

監査結果は次の順で重視する。

1. 外部攻撃面や権限乱用に直結する欠陥
2. 機密情報・個人情報・認証情報の保護不備
3. 監査ログ、変更管理、バックアップ、復旧など継続運用リスク
4. 文書化不足や運用統制不足による監査不能リスク

## 出力要件

出力はMarkdownで、最低限次の見出しを含める。

- `監査概要`
- `監査対象と前提`
- `総合評価`
- `重要指摘`
- `監査結果一覧`
- `証跡と根拠`
- `未確認事項`
- `優先改善計画`
- `適用基準`

追加ルール:

- `重要指摘` は `Critical` / `High` / `Medium` / `Low` の重大度を使う。
- `監査結果一覧` は最低でも `監査項目 | 判定 | 重大度 | 根拠 | 推奨対応` を含める。
- `combined` の場合は `system` と `governance` を章分けし、先頭に統合総評を置く。
- 未確認事項が多い場合は、「監査可能性不足」自体を総合評価に含める。

## 監査時の着眼点

### `system` モード

優先して読む対象:

- アプリケーションコード
- 認証認可設定
- シークレット管理設定
- 依存関係定義
- IaC
- CI/CD設定
- ログ、監視、バックアップ設定
- `references/corpus/*.toon`

### `governance` モード

優先して読む対象:

- セキュリティ方針
- 権限申請や棚卸手順
- インシデント対応手順
- 変更管理手順
- 委託先管理資料
- 教育や訓練の記録
- 監査証跡の保存方針
- `references/corpus/*.toon`

## 品質チェック

- すべての重要指摘に根拠がある。
- 根拠がない項目を `適合` にしていない。
- 未確認事項と改善提案を混同していない。
- 質問は監査項目に紐づいている。
- 監査対象の前提が冒頭に明示されている。
- framework別根拠の表記が混在していない。
