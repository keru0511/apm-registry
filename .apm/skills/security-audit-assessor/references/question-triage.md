# 不足情報の質問基準

質問は、既存資料を読んでも重大な判定ができない場合だけ行う。最大3問ずつに制限する。

## 質問が必要な条件

次のいずれかに当てはまる場合は質問してよい。

- `Critical` または `High` 相当の項目が、対象条件不明のため判定不能
- 監査モードの選択に必要な情報が欠けている
- データ分類、公開範囲、認証方式、責任分界のような前提不明で複数項目が連鎖的に判定不能

## 質問しない条件

- コードや設定を読めば分かる
- 推奨改善には影響するが、現状判定には影響しない
- `要確認` として残しても監査結果の意味が損なわれない

## 優先して聞くべき情報

1. 外部公開範囲
   - 例: 本番環境はインターネット公開か、社内限定か
   - 影響: ネットワーク境界、WAF、監視、ログ
2. データ分類
   - 例: 個人情報、決済情報、機密情報を扱うか
   - 影響: データ保護、暗号化、保持、削除、権限制御
3. 認証方式と利用者種別
   - 例: 社員向けのみか、顧客向けか、SSOか、ローカルIDか
   - 影響: 認証認可、MFA、棚卸、監査証跡
4. 運用責任分界
   - 例: クラウド事業者、委託先、自社運用の分担
   - 影響: 委託先管理、監視、インシデント対応、変更管理
5. 復旧要件
   - 例: RPO/RTO、バックアップ頻度、復旧試験の有無
   - 影響: 可用性、継続計画、復旧統制

## 質問文の形式

各質問は次の形にする。

- 質問: `...`
- 影響項目: `監査項目A, 監査項目B`
- 未回答時の扱い: `要確認のまま報告する`

## 1ラウンドの制限

- 1ラウンド最大3問
- 3問を超える場合は、影響範囲が最も広いものから優先する
- 回答後に残件があれば、必要なときだけ次ラウンドを出す
