# 不足情報の質問基準

質問は、既存資料を読んでも監査に必要な事実が確定できない場合だけ行う。質問数に固定上限は置かず、判定に必要な最小限を段階的に聞く。

## 質問が必要な条件

次のいずれかに当てはまる場合は質問してよい。

- リポジトリ、設定、IaC、README、運用文書に存在しない事実が、監査結論や重要な推奨対応に影響する
- 監査モードの選択に必要な情報が欠けている
- データ分類、公開範囲、認証方式、責任分界のような前提不明で複数項目が連鎖的に判定不能
- 既存資料に矛盾があり、どの前提を採用すべきか確定できない

## 質問しない条件

- コードや設定を読めば分かる
- README や運用文書に明記されている
- 推奨改善には影響するが、現状判定には影響しない
- `要確認` として残しても監査結果の意味が損なわれない

## 優先して聞くべき情報

1. 外部公開範囲
   - 例: 本番環境はインターネット公開か、社内限定か
   - 影響: ネットワーク境界、WAF、監視、ログ
2. データ分類
   - 例: 個人情報、決済情報、機密情報を扱うか
   - 影響: データ保護、暗号化、保持、削除、権限制御
3. 認証方式と利用者種別
   - 例: 社員向けのみか、顧客向けか、SSOか、ローカルIDか
   - 影響: 認証認可、MFA、棚卸、監査証跡
4. 運用責任分界
   - 例: クラウド事業者、委託先、自社運用の分担
   - 影響: 委託先管理、監視、インシデント対応、変更管理
5. 復旧要件
   - 例: RPO/RTO、バックアップ頻度、復旧試験の有無
   - 影響: 可用性、継続計画、復旧統制

## 質問文の形式

各質問は次の形にする。

- 暫定前提: `現時点では ... と仮置き`
- 質問: `...`
- 選択肢:
  - `1. ...`
  - `2. ...`
  - `3. ...`
  - `4. ...`
  - `5. 不明/未確認`
- 影響項目: `監査項目A, 監査項目B`
- 未回答時の扱い: `要確認のまま報告する`

## 選択式で聞く原則

- 原則として、ユーザーは番号または選択肢名を返すだけでよい形にする。
- 選択肢は `2-5個 + 不明/未確認` を基本とし、長い自由記述を求めない。
- 選択肢は監査観点ごとに粗すぎず細かすぎない粒度にまとめる。
- 既存資料から最もありそうな前提があれば、それを `暫定前提` として先に示す。
- 自由入力が必要なのは、環境固有の責任分界や例外運用など、選択肢化すると誤判定リスクが高い場合だけにする。

## 段階式で出す原則

- 最初に対象資料を読み、確定できる前提を埋める。
- 質問は、影響範囲の広い前提から順に出す。
- 回答によって不要になった質問は出さない。
- 1回の提示は短く保ち、監査継続に必要な最小限だけを聞く。
- 一括で長い質問票を並べるより、判定に必要な順で分けて聞く。

## 出し方の例

- 暫定前提: `README と構成からは API が社内向けの可能性が高い`
- 質問: `本番公開範囲はどれですか`
- 選択肢: `1. インターネット公開 2. 社内/VPN限定 3. 未本番化 4. 不明`
- 影響項目: `ネットワーク境界, WAF, 監視, 公開面管理`
- 未回答時の扱い: `要確認のまま報告する`

- 暫定前提: `アプリ設定からは SSO 連携が見当たらない`
- 質問: `利用者と認証方式はどれに近いですか`
- 選択肢: `1. 社員向けSSO 2. 社員向けローカルID 3. 顧客向け認証 4. 不明`
- 影響項目: `認証認可, MFA, アカウント棚卸, 監査証跡`
- 未回答時の扱い: `要確認のまま報告する`
