# 比較提案資料作成スキル

入力条件が曖昧でも停止せず、明示した前提を置いて比較提案資料を完成させる。
意思決定者向けの提案資料を作る場合は、比較表より先に「結論・決裁事項」を提示する。

## 参照ファイル

- 公的資料要約一覧: `references/public-standards-summary.md`
- 標準テンプレート: `references/comparison-proposal-template.md`
- 評価軸カタログ: `references/evaluation-criteria-catalog.md`

最初に `public-standards-summary.md` だけを読み、案件に必要な資料を特定する。
その後に必要最小限で他の参照ファイルを読み、目的に不要な項目は削る。

## 進め方

1. 比較の目的を1文で定義する。
2. 比較対象を列挙し、比較対象ごとの前提条件をそろえる。
3. 評価軸を選定し、重み（合計100%）を決める。
4. 各評価軸を5段階で採点し、根拠を短く記載する。
5. 重み付き合計点を算出し、順位を出す。
6. 点数差だけでなく、主要なトレードオフを文章で明示する。
7. 推奨案を1つ選び、採用理由・不採用理由・導入計画を記載する。
8. 未確定事項、追加調査項目、意思決定期限を明記する。

## 提案資料モード（既定）

「提案資料」「提案書」「意思決定用」と依頼された場合は、次の順番で出力する。

1. 提案要旨（結論先出し）
2. 背景と課題
3. 選定方針と評価観点
4. 比較サマリー（1表）
5. 推奨案の詳細（採用理由・不採用理由）
6. 導入計画（30/60/90日）
7. リスクと対策
8. 意思決定依頼（承認してほしい事項）

比較点数の詳細表は補足扱いとし、本文は意思決定に必要な情報を優先する。

## 解釈ルール（曖昧さ防止）

- 「本文先頭」の定義:
  - `提案要旨` は最初の見出しで配置する。
  - `意思決定依頼` は `提案要旨` 内、または本文の3見出し以内に配置する。
- 参照ファイルの必須度:
  - `public-standards-summary.md` は必須で読む。
  - `comparison-proposal-template.md` と `evaluation-criteria-catalog.md` は案件に必要な場合のみ読む。
- 補足表の必須条件:
  - 「比較詳細（重み付き）」は既定では任意。
  - 「点数根拠の提示」「詳細スコア提出」が依頼された場合のみ必須化する。

## スコアリング規則

- 評点は `1` から `5` の整数を使う。
- 重み付き点は `評点 / 5 * 重み(%)` で計算する。
- 合計点は `0` から `100` の範囲で示す。
- 定量根拠がない項目は「暫定評価」と明記する。
- 「暫定評価」の付与基準:
  - 公式資料・一次情報で未検証の数値を使う場合
  - 仮定した前提（人数、予算、期限、為替、契約条件）に依存する場合
  - 比較対象の仕様や名称が確定していない場合

## 出力要件

- 出力形式はMarkdown。
- 提案資料モードでは最低限、次の見出しを含める:
  - 提案要旨（Executive Summary）
  - 背景と課題
  - 比較サマリー
  - 推奨案と理由
  - 導入計画（30/60/90日）
  - リスクと対策
  - 意思決定依頼
- 比較詳細は補足として末尾に配置し、本文の先頭に置かない。
- 最後に「意思決定者が今すぐ判断できるか」を1文で評価する。
- 断定表現には根拠を添える。根拠が弱い場合は「暫定評価」と明記する。

## 品質チェック

- 評価軸が目的に対して過不足ない。
- 重みが経営・現場の優先度を反映している。
- 採点根拠が主観だけで終わっていない。
- 推奨案の弱点と回避策が書かれている。
- 「次に誰が何をいつまでに実施するか」が具体化されている。
- 本文冒頭で、結論と承認依頼事項が明確になっている。
